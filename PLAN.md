# Solo-план разработки мессенджера (Android + ПК + Node.js)

Этот документ превращает общий roadmap в **конкретные задачи**, чтобы по ним можно было идти шаг за шагом одному разработчику.

## 0) Правила выполнения

- Работать короткими итерациями (1 неделя = 1 демонстрируемый результат).
- Каждая задача должна иметь: результат, критерий готовности, артефакт.
- Приоритет: `MVP` → `Стабильность` → `Полировка`.

---

## 1) Недели 1–2: Основа проекта

### 1.1 Монорепа и базовая инфраструктура
- [ ] Создать структуру репозитория:
  - `apps/server`
  - `apps/web`
  - `apps/mobile`
  - `packages/shared-types`
- [ ] Настроить единый `TypeScript` и линтинг.
- [ ] Поднять `docker-compose` для PostgreSQL + Redis + MinIO.
- [ ] Добавить `.env.example` для всех приложений.

**Критерий готовности:**
- Все сервисы стартуют одной командой, локальная среда воспроизводима.

### 1.2 Backend: Auth (Node.js + NestJS)
- [ ] Инициализировать NestJS-проект.
- [ ] Модели БД: `users`, `refresh_tokens`, `devices`.
- [ ] Реализовать API:
  - `POST /auth/register`
  - `POST /auth/login`
  - `POST /auth/refresh`
  - `POST /auth/logout`
- [ ] Хеширование паролей (`argon2`/`bcrypt`).
- [ ] JWT access + refresh с ротацией.

**Критерий готовности:**
- Можно зарегистрироваться, войти, обновить токен и выйти.

### 1.3 UI-фундамент и тёмная тема
- [ ] Создать дизайн-токены (цвета, шрифты, отступы, радиусы).
- [ ] Реализовать базовые UI-компоненты:
  - Button
  - Input
  - Card
  - Avatar
  - Loader
- [ ] Экран логина/регистрации для web и mobile.

**Критерий готовности:**
- Логин и регистрация работают в тёмной теме на обеих платформах.

---

## 2) Недели 3–4: Чаты 1:1

### 2.1 Модель и API чатов
- [ ] Таблицы: `chats`, `chat_members`, `messages`.
- [ ] API:
  - `POST /chats/direct`
  - `GET /chats`
  - `GET /chats/:id/messages`
  - `POST /chats/:id/messages`
- [ ] Пагинация сообщений (cursor-based).

**Критерий готовности:**
- Два пользователя могут создать диалог и увидеть историю.

### 2.2 Realtime через Socket.IO
- [ ] Подключение клиента по JWT.
- [ ] События:
  - `message:new`
  - `message:ack`
- [ ] Гарантия, что отправленное сообщение появляется у собеседника без перезагрузки.

**Критерий готовности:**
- Сообщения приходят в реальном времени между web/mobile клиентами.

### 2.3 Клиентские экраны
- [ ] Экран списка чатов.
- [ ] Экран диалога.
- [ ] Поле ввода + отправка сообщений.

**Критерий готовности:**
- Базовая переписка 1:1 работает end-to-end.

---

## 3) Недели 5–6: Качество общения

### 3.1 Presence и typing
- [ ] Реализовать online/offline (Redis + heartbeat).
- [ ] События `typing:start` / `typing:stop`.

**Критерий готовности:**
- В диалоге видно, когда пользователь онлайн и печатает.

### 3.2 Статусы доставки и прочтения
- [ ] Таблица `message_reads`.
- [ ] Статусы: `sent`, `delivered`, `read`.
- [ ] Обновление статусов в реальном времени.

**Критерий готовности:**
- У каждого сообщения корректно отображается статус.

### 3.3 Поиск
- [ ] Поиск пользователей по имени/логину.
- [ ] Поиск чатов по названию/участнику.

**Критерий готовности:**
- Пользователь находит нужный чат/контакт за 1–2 запроса.

---

## 4) Недели 7–8: Медиа и уведомления

### 4.1 Вложения
- [ ] Таблица `attachments`.
- [ ] Загрузка в MinIO/S3.
- [ ] Привязка вложений к сообщениям.
- [ ] Ограничения по размеру и типам файлов.

**Критерий готовности:**
- Можно отправить изображение/файл и открыть его у собеседника.

### 4.2 Уведомления
- [ ] Android push (FCM).
- [ ] Desktop notifications (Electron/Web).
- [ ] Настройки уведомлений в профиле.

**Критерий готовности:**
- При новом сообщении пользователь получает уведомление вне активного окна чата.

### 4.3 Надёжность отправки
- [ ] Retry-логика при сетевых сбоях.
- [ ] Локальный признак «не отправлено», кнопка повторной отправки.

**Критерий готовности:**
- Потеря сети не ломает диалог и не теряет сообщения на клиенте.

---

## 5) Недели 9–10: Desktop + безопасность

### 5.1 Desktop-сборка
- [ ] Упаковать web-клиент в Electron.
- [ ] Добавить окружения dev/prod.
- [ ] Подготовить сборки для Windows/macOS/Linux.

**Критерий готовности:**
- Приложение устанавливается и запускается локально как desktop client.

### 5.2 Усиление безопасности
- [ ] `helmet`, `cors`, rate limit.
- [ ] DTO-валидация входных данных.
- [ ] Логи аудита входов/ошибок.

**Критерий готовности:**
- Базовые атаки (bruteforce/flood) ограничены.

### 5.3 Производительность
- [ ] Виртуализация длинных списков сообщений.
- [ ] Индексы БД для основных запросов.
- [ ] Профилирование горячих endpoint'ов.

**Критерий готовности:**
- Диалог с 10k+ сообщений остаётся отзывчивым.

---

## 6) Недели 11–12: Полировка и релиз MVP

### 6.1 UX-полировка
- [ ] Контраст и доступность (минимум AA для текста).
- [ ] Анимации переходов (150–250ms).
- [ ] Состояния ошибок/пустых экранов/загрузки.

**Критерий готовности:**
- Интерфейс визуально целостный и предсказуемый.

### 6.2 Наблюдаемость и эксплуатация
- [ ] Подключить Sentry.
- [ ] Метрики + базовые алерты (CPU, RAM, 5xx).
- [ ] Бэкап PostgreSQL + тест восстановления.

**Критерий готовности:**
- Есть базовая эксплуатационная готовность к первым пользователям.

### 6.3 Релиз
- [ ] Release checklist.
- [ ] Smoke-тест по критическим сценариям.
- [ ] Публикация MVP-сборок.

**Критерий готовности:**
- MVP готов к закрытому тесту (pilot users).

---

## 7) Ежедневный solo-ритм (операционный чеклист)

Каждый рабочий день:
- [ ] 1 ключевая продуктовая задача.
- [ ] 1 улучшение UX/UI.
- [ ] 1 задача по стабильности (лог/валидация/тест).
- [ ] Запись в `dev-log`:
  - Что сделано
  - Что блокирует
  - План на завтра

---

## 8) Definition of Done (для любой задачи)

Задача считается завершённой только если:
- [ ] Код в репозитории и проходит линт/тесты.
- [ ] Добавлены/обновлены миграции и документация API (если нужно).
- [ ] Проверен happy path и минимум 1 error path.
- [ ] Нет критичных регрессий на web/mobile.

---

## 9) Что не входит в MVP (осознанно)

- [ ] Групповые чаты с ролями и модерацией.
- [ ] Аудио/видео звонки (WebRTC).
- [ ] End-to-end шифрование.
- [ ] Полноценная админ-панель аналитики.

Эти пункты переносятся в Phase 2 после обратной связи от первых пользователей.
